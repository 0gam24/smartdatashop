## 1. 핵심 주장

> **"하네스 엔지니어링은 일급 엔지니어링이다. CLAUDE.md는 AI의 행동을 결정하는 코드다."**

**자동차 와이어링 하네스 비유**: 수백 개의 전선을 정리하는 시스템처럼, AI에 흐르는 4가지 신호를 코드/설정 파일로 체계화.

```
Agent = Model + Harness
```

**증명**: OpenAI Codex 100만 줄 / LangChain Terminal Bench 30→5위 / Theo: Cursor +20% / Mitchell Hashimoto: "AI가 실수하면, 구조적으로 그 실수를 반복할 수 없게 만든다"

## 2. 안티패턴 7가지

| 코드 | 패턴 | 위험 |
|---|---|---|
| **A1** | 프롬프트만으로 AI 제어 | 지속성 X / 일관성 X / 강제성 X |
| **A2** | 코드 작성과 리뷰를 같은 에이전트가 | 자기 편향 발생 |
| **A3** | 모든 에이전트에 같은 컨텍스트 부여 | 컨텍스트 낭비 |
| **A4** | settings.local.json을 .gitignore 안 함 | 개인 설정이 팀 설정 오염 |
| **A5** | CLAUDE.md 변경을 일반 코드 PR과 다른 취급 | AI 행동 결정 코드인데 리뷰 없이 머지 |
| **A6** | 규칙에 이유 안 남김 | 시간 지나면 의문만 남고 무시되거나 잘못 제거됨 |
| **A7** | 수십 규칙 정리 안 함 | 하네스가 비대해지고 규칙 간 충돌 |

## 3. 적용 가능한 패턴 9가지

### P1. 하네스의 4가지 신호 체계 ★
1. **지침** (코딩 컨벤션, 기술 스택 규칙) → CLAUDE.md / AGENTS.md
2. **제약** (파일 수정 전 자동 검사) → Hook
3. **자동화** (반복 워크플로우 표준 절차) → Skills
4. **도구 접근** (외부 서비스 연결) → MCP

### P2. 멀티 에이전트 3차원
| 차원 | 설명 | 예시 |
|---|---|---|
| Sub-agent | Task 도구로 생성된 하위 에이전트 | 리서치 / 검증 / 변환 에이전트 |
| Agent Teams | 같은 프로젝트 여러 Claude Code 인스턴스 | 터미널 탭별 다른 사이트 작업 |
| 팀 에이전트 | 여러 명이 같은 코드베이스 기여 | (단독 운영자 해당 없음) |

### P3. 5가지 멀티 에이전트 패턴 ★
1. **작성 vs 리뷰 분리** — 자기 편향 방지
2. **프론트엔드 vs 백엔드** — 디렉토리별 CLAUDE.md
3. **탐색 vs 구현 분리** — 컨텍스트 절약 (가장 자주 활용)
4. **사이트별 병렬** — Agent Teams로 터미널 병렬
5. **Sub-agent 호출 4요소**: 역할 / 범위 / 제약 / 출력 형식

### P4. CLAUDE.md 설정 구조
```
~/.claude/CLAUDE.md         — 사용자 전체 (별도 관리)
└── 프로젝트/CLAUDE.md       — 팀 공통 (Git 커밋)
    ├── frontend/CLAUDE.md   — 영역별 (Git 커밋)
    └── backend/CLAUDE.md    — 영역별 (Git 커밋)
```
Sub-agent도 모든 계층의 규칙을 자동 상속받음.

### P5. 5계층 설정 분리 ★★★
| 파일 | 범위 | Git |
|---|---|---|
| `CLAUDE.md` / `AGENTS.md` | 프로젝트 전체 | 커밋 |
| `[디렉토리]/CLAUDE.md` | 디렉토리 | 커밋 |
| `.claude/settings.json` | 프로젝트 | 커밋 |
| `.claude/settings.local.json` | 개인 | **.gitignore 필수** |
| `~/.claude/CLAUDE.md` | 사용자 전체 | 별도 |

### P6. Hook 3종 패턴
- **PostToolUse**: 파일 수정 직후 (예: prettier 자동 포맷)
- **SubagentStop**: sub-agent 작업 종료 시 (예: lint 검증)
- **Stop**: 전체 세션 종료 시 (예: 빌드 검증)

### P7. "하네스도 코드다" 운영 원칙
- Git 버전 관리 + PR 리뷰 + 변경 이력 추적 필수
- **제 1회 정기 리뷰**: 무효 규칙 제거 / 무시되는 규칙 원인 분석 / 신 패턴 추가 / Hook 효과 측정

### P8. 규칙에 이유 주석
```markdown
## 금지 사항
<!-- Cloudflare Pages는 Node.js API를 지원하지 않으므로 -->
- fs, path 등 Node.js 내장 모듈을 클라이언트 코드에서 사용하지 않는다

<!-- 2026-03 빌드 사고 이후 추가 -->
- 프로덕션 빌드 전 반드시 npm run check 통과
```

## 4. smartdatashop.kr 즉시 적용 항목

### 이미 적용된 부분
- ✅ `.claude/` 폴더 존재
- ✅ `docs/AGENTS.md` 존재 (이외사 안내문)
- ✅ 6 에이전트 stub (멀티 에이전트 P2의 Sub-agent 활용 시작점)

### 추가로 가치 있는 것

#### 1. **6 에이전트 stub을 P5 5계층으로 정리** (P4, P5)
현재 stub들이 어디에 위치하는지 / 어떤 디렉토리 규칙을 상속하는지 명확히.

#### 2. **Sub-agent 호출 4요소 표준화** (P3, P5)
앞으로 Sub-agent에 작업 의뢰할 때 항상 4요소 명시:
- 30~50편 펄스 검수 → "검수 에이전트, content/pulse만, 수정 금지, 검수 결과 표"
- SEO 감사 → "SEO 에이전트, 모든 페이지, 코드 X, 문제 목록"
- 새 카테고리 추가 → "구현 에이전트, src/와 content/ 가능, 기존 디자인 시스템 준수, 변경 파일 목록"

#### 3. **Hook 3종 시작 검토** (P6)
- PostToolUse: prettier (Astro+Tailwind 환경에 맞춰 설정)
- SubagentStop: lint
- Stop: `npm run build` 검증

> 단, **이미 .claude/settings.json이 있을 가능성** — 부서지지 않도록 누락된 Hook만 추가

#### 4. **규칙에 이유 주석 다음** (P8)
docs/AGENTS.md의 모든 규칙에 `<!-- 왜 이 규칙인가 -->` 주석 추가.
6개월 후 자기 자신이 의문 안 가지도록.

#### 5. **제 1회 하네스 정기 리뷰** (P7)
