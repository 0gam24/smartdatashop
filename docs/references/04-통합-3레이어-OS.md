# 📘 통합 레퍼런스 — 서로 이어지는 3-레이어 OS — smartdatashop.kr 적용

> **출처**: 위키독스 책 3권 분석의 합성
> **분류**: 통합 — 큰 그림 의사결정 가이드
> **smartdatashop.kr 적용 영역**: 큰 결정 / 책 3권 중 어떤 패턴을 적용할지 갈등 시

## 1. 핵심 통찰 — 3권의 책이 한 골격의 다른 각도

세 책의 패턴이 합쳐지면 서로 이어지는 완전한 OS가 만들어짐:

```
┌─────────────────────────────────────────┐
│ [외부 레이어]  책 19307                   │
│ "이전 메타 시스템"                        │
│ → 다중 프로젝트 / 노트-First / 시간 예산  │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│ [실행 레이어]  책 19303                   │
│ "단일 프로젝트 라이프사이클"              │
│ → PRD → 설계 → 구현 → 테스트 → 배포      │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│ [제어 레이어]  책 19470                   │
│ "AI 행동 자동 제어"                       │
│ → CLAUDE.md + Hook + 멀티 에이전트       │
└─────────────────────────────────────────┘
```

**한 줄 요약**: 외부에서는 "어디로 갈지", 실행에서는 "어떻게 갈지", 제어에서는 "AI가 길을 벗어나지 않게" 관리.

## 2. smartdatashop.kr에 매핑

### 외부 레이어 — 다중 사이트 운영 (자매 5개 합류 시 본격 활용)
**현재**: 메인 1 + 자매 4 (기존 도메인 유지) 운영 중
**M3~M12**: 자매 5개 신규 합류 (homedata, bizdata, familydata, retiredata, aidata)

이 레이어가 본격적으로 가치 있어지는 시점은 **자매 5개가 추가되어 총 10개 사이트 운영할 때**.

적용 패턴:
- 노트-First (책 19307 P1)
- 자기완결 폴더 구조 (각 사이트 docs/ 일관성)
- 공통 폴더 = 자매 사이트 횡단 지식
- 시간 예산 (30초/3분/5분 룰)
- 대시보드 1 페이지

### 실행 레이어 — 단일 사이트 라이프사이클 (이미 활용 중)
**smartdatashop.kr 현재 상태**: 5단계 모두 통과, 가동 중

이 레이어를 다시 적용해야 할 때:
- 자매 사이트 1개 신규 추가 (homedata.kr 등)
- 새 가이드북 권 추가 (3권, 4권...)
- 새 카테고리 신설 (예: "스타트업·창업" 6번째 카테고리)

적용 패턴:
- 5단계 로드맵 (책 19303 P1)
- 모듈 분리 1줄 기준 (P2)
- SDD (P4)
- 컨텍스트 엔지니어링 (P5) — 일상 작업에서도 가치
- Context Maps (P6)

### 제어 레이어 — AI 행동 자동 제어 (일상 활용)
**smartdatashop.kr 현재**: docs/AGENTS.md, .claude/, 6 에이전트 stub 존재

이 레이어는 **매일 활용**:
- 바이브 코딩 시작할 때마다 AGENTS.md가 자동 로드
- Sub-agent 작업 시 4요소 패턴
- Hook으로 자동 검증
- 규칙에 이유 주석

## 3. 책 3권 패턴 통합 매트릭스

같은 문제가 서로 다른 책에서 다른 각도로 해결됨:

| 문제 | 책 19307 (외부) | 책 19303 (실행) | 책 19470 (제어) |
|---|---|---|---|
| AI가 같은 실수 반복 | 에러로그 자산화 | 룰 파일에 반영 | CLAUDE.md + Hook으로 강제 |
| 큰 작업 시 컨텍스트 폭발 | 노트-First | 컨텍스트 엔지니어링 | 탐색 vs 구현 분리 |
| 다중 작업 컨텍스트 전환 | 자기완결 폴더 + 대시보드 | (직접 다루지 않음) | Agent Teams 패턴 |
| 규칙이 흐려지거나 무시됨 | 의사결정 발자국 (ADR) | (직접 다루지 않음) | 규칙 이유 주석 + 정기 리뷰 |
| 검토 없이 무한 승인 | "Accept All" 안티패턴 | 모듈 분리로 작업 단위 작게 | Hook으로 게이트 |
| 폴더 구조 혼란 | 자기완결 폴더 | architecture.md | 디렉토리별 CLAUDE.md |

**활용 방법**: 어떤 문제를 만났을 때 위 표을 보고 3가지 각도의 해법 중 가장 적합한 것 선택.

## 5. 핵심 인용 — 자주 떠 올릴 5가지

### Karpathy
> "LLM이 CPU라면, 컨텍스트 윈도우는 RAM이다."
→ 큰 작업 시작 전 컨텍스트 설계가 필수.

### 안트로비
> "바이브 코딩의 성공은 코딩 실력이 아니라 논리적 설계 능력에 달려 있다."
→ 막혔을 때 코딩 더 하기 전에 설계로 돌아갈 것.

### 안트로비
> "설계에 1시간 투자 → 구현에서 10시간 절약."
→ 설계가 답답하게 느껴져도 그게 가장 빠른 길.

### 함께해요 책
> **Agent = Model + Harness**
→ AI 성능은 모델만의 문제가 아니라 통제 시스템의 문제.

### Mitchell Hashimoto
> "AI가 실수하면, 구조적으로 그 실수를 반복할 수 없게 만든다."
→ 같은 실수 두 번째 = 즉시 CLAUDE.md / Hook으로 반영.
