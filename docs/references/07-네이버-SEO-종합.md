## 2. 네이버 검색 생태계 — 어디에 노출되는가

네이버 검색 결과는 Google과 다르게 **여러 컬렉션이 영역별로 구분**되어 표시된다.

### 2.1 주요 노출 영역 (2026년 5월 기준)

| 영역 | 출처 | 외부 사이트 노출 가능? | 우리 사이트 적용 |
|---|---|---|---|
| **파워링크/광고** | 네이버 검색광고 | 유료 | 해당 없음 |
| **VIEW (스마트블록 통합)** | 네이버 블로그, 카페, 포스트 | ❌ 외부 사이트 불가 | 별도 블로그 운영 시 |
| **인플루언서 검색** | 인플루언서로 등록된 블로거·유튜버 | ❌ | 해당 없음 |
| **웹사이트** (구 웹문서) | **외부 사이트 일반 웹페이지** | ✅ **YES** | **smartdatashop.kr 메인 노출 영역** |
| **뉴스** | 네이버 뉴스 제휴 매체 | ❌ 일반 매체는 제휴 필요 | 해당 없음 |
| **쇼핑** | 네이버 스마트스토어, 외부 쇼핑몰 | 일부 가능 | 해당 없음 |
| **어학사전·지식백과·지식iN** | 자체 DB 또는 사용자 Q&A | ❌ | 해당 없음 |
| **이미지·동영상** | 외부 사이트 + 네이버 자체 | ✅ 부분 가능 | 이미지 사이트맵·동영상 사이트맵 |
| **카페** | 네이버 카페 | ❌ | 해당 없음 |

### 2.2 외부 사이트 운영자가 집중할 영역

> **smartdatashop.kr 같은 외부 사이트는 "웹사이트" 영역**이 핵심 노출 채널이다. 이 영역은 다음의 영향받는다:

- **출처 신뢰도** (C-Rank 일부 적용)
- **페이지의 정보 품질** (D.I.A.)
- **검색 의도 일치도** (D.I.A.+)
- **백링크** (콘텐츠 확산)
- **HTML 마크업 품질** (사이트 진단)
- **모바일 최적화**
- **HTTPS**
- **사이트 응답 속도**

### 2.3 스마트블록 시대의 변화

2025년 이후 네이버는 검색 결과를 **스마트블록**(주제별 블록 단위 노출)으로 통합한다. 이전에 별도였던 VIEW·웹사이트·이미지·동영상이 한 블록 안에 섞여 표시되기도 한다.

영향:
- 단일 키워드의 1위 자리는 줄어듦, 대신 **다양한 출처가 노출 기회를 얻음**
- 블록의 주제에 정확히 부합하는 콘텐츠가 우대됨
- 외부 사이트는 **세부 주제 + 깊이**가 있으면 블록 진입 가능

## 3. 네이버 검색 알고리즘

### 3.1 C-Rank (씨랭크) — 출처(채널) 신뢰도

> **블로그·카페·포스트** 영역에서 주로 작용하지만, 네이버 검색 전반의 출처 평가에 영향을 준다.

C-Rank의 세 축 — **C**ontext, **C**ontent, **C**hain

| 신호 | 측정 대상 |
|---|---|
| **BLOG ID** | 출처(블로그) 자체의 정체성·일관성 |
| **BLOG 인기도** | 방문자 수, 활성 사용자 수 |
| **Search LOG** | 검색 사용자의 클릭·체류·재방문 패턴 |
| **Chain Score** | 외부에서의 인용·링크·언급 (백링크 + 언급) |
| **Blog Activity** | 발행 빈도, 편집 활동 |
| **Blog Editor 주제 점수** | 딥러닝으로 분류한 주제의 집중도 |

**핵심 원리**: "한 주제에 얼마나 깊이 있는 좋은 콘텐츠를 꾸준히 생산하는가"

> **smartdatashop.kr 적용**: 5개 카테고리(정책/세금·금융/시장/통계/AI·기술)에 집중 + 매일 발행 + 1차 출처 + verifiedBy = C-Rank 우호적 시그널.
>
> **자매 사이트 적용**: 각 사이트가 단일 페르소나·카테고리에 집중할수록 C-Rank가 강화된다. **동일 주제를 9개 사이트에 분산하면 각각의 C-Rank가 약해질 수 있다** → 페르소나 차별화가 필수.

### 3.2 D.I.A. (다이아) — 문서 정보성

**D**eep **I**ntent **A**nalysis. 문서 단위로 정보성·경험·만족도를 평가한다.

C-Rank가 출처(누가 썼냐)에 가중치를 둔다면, D.I.A.는 **문서 자체의 품질**(무엇을 썼냐)에 집중한다.

평가 대상 (네이버 비공개 + 추정):
- 사용자의 **체류 시간**
- **공유 횟수**
- **댓글 내용의 진정성**
- **재방문율**
- **문서의 정보 풍부도**
- **저자의 직접 경험 흔적** (직접 사용·방문·체험)

### 3.3 D.I.A.+ (다이아 플러스) — 검색 의도 분석

D.I.A.를 확장하여 **사용자의 검색 의도를 더 깊이 분석**한다.

예시:
- "강남역 스테이크 맛집 추천" → 단순 키워드 일치가 아닌, 실제 방문 후기·가격대·분위기·예약 정보 등 **사용자 의도를 충족하는 정보 깊이**를 평가

### 3.4 핵심 시사점

세 알고리즘이 함께 작동한다:
- **C-Rank**: 사이트(출처) 점수
- **D.I.A.**: 페이지(문서) 점수
- **D.I.A.+**: 검색어와 페이지의 의도 일치도

세 점수가 결합되어 최종 순위가 결정된다.

> **smartdatashop.kr 적용**:
> - 출처: 메인은 광고 없는 신뢰 자산 모델 → 출처 신뢰도 강화
> - 문서: 1차 출처 + 운영자 검토 + verifiedBy → D.I.A. 우호적
> - 의도: "한국의 데이터를 매일 5분으로" 포지션 → 사용자 검색 의도(빠르고 정확한 정보)에 정렬

### 3.5 Google E-E-A-T와의 비교

| Google E-E-A-T | 네이버 알고리즘 대응 |
|---|---|
| Experience (경험) | D.I.A. (문서의 직접 경험 흔적) |
| Expertise (전문성) | C-Rank (주제 집중도) |
| Authoritativeness (권위) | C-Rank (Chain Score, 백링크) |
| Trustworthiness (신뢰성) | C-Rank + D.I.A. (출처+문서 모두) |

→ **Google E-E-A-T를 충실히 따르면 네이버 알고리즘에도 자연스럽게 부합**한다.

## 4. 네이버 검색봇 Yeti

### 4.1 Yeti 기본 정보

| 항목 | 값 |
|---|---|
| 정식 명칭 | Yeti |
| User-agent (PC) | `Mozilla/5.0 (compatible; Yeti/1.1; +https://naver.me/spd)` |
| User-agent (모바일) | `Mozilla/5.0 (Linux; U; Android ...) ... Mobile Yeti` |
| IP 대역 | 네이버가 공식 페이지에서 공개 |
| 방문 빈도 | 등록 사이트 기준 **하루 1회 이상** |

### 4.2 robots.txt에서 Yeti 처리

**모든 검색엔진 허용 (smartdatashop.kr 권장 기본)**:
```
User-agent: *
Allow: /

Sitemap: https://smartdatashop.kr/sitemap-index.xml
```

**Yeti 전용 규칙 (특정 디렉토리 차단)**:
```
User-agent: Yeti
Disallow: /admin/
Disallow: /api/
Allow: /

User-agent: *
Allow: /

Sitemap: https://smartdatashop.kr/sitemap-index.xml
```

**모바일 Yeti는 별도 처리하려면**:
```
User-agent: Yeti
Disallow: /private/

User-agent: Yeti-Mobile
Disallow: /pc-only/
```

### 4.3 검색봇 확인 — 위장 봇 식별

네이버는 Yeti의 **공식 IP 대역**을 공개한다. 서버 로그에서 `Yeti` User-agent로 들어왔는데 IP가 공개 대역 외라면 **위장 봇** 가능성이 높다.

서치어드바이저 > **검증 > 검색로봇 확인** 메뉴에서 IP를 입력해 검증할 수 있다.

## 6. 사이트 등록 및 소유권 확인

### 6.1 등록 절차

1. **네이버 계정 로그인** — searchadvisor.naver.com (네이버 계정 그대로 사용)
2. **웹마스터 도구 진입**
3. **사이트 추가** — `https://smartdatashop.kr` 형식 최상위 URL 입력
4. **소유권 확인**:
   - **HTML 파일 업로드**: 네이버 제공 HTML 파일을 사이트 루트에 업로드
   - **HTML 태그**: `<head>` 안에 메타 태그 추가

#### HTML 태그 방식 예시

```html
<meta name="naver-site-verification" content="abc123def456ghi789..." />
```

### 6.2 도메인 단위 등록의 함의

네이버는 **도메인 단위로 사이트를 등록**한다. 즉:
- `smartdatashop.kr` 등록 시 → `https://smartdatashop.kr/...`의 모든 URL이 대상
- `homedata.kr`, `bizdata.kr` 등 자매 사이트는 **각각 별도로 등록**해야 한다
- `www.smartdatashop.kr` vs `smartdatashop.kr`는 다르게 인식될 수 있음 → **301 리디렉트**으로 한쪽으로 통일 필수

### 6.3 등록 후 대기 시간

- 소유권 확인: 보통 즉시 ~ 몇 시간 (글이 없으면 더 오래)
- 첫 인덱스 등록: **약 14-16일**
- 새 콘텐츠 인덱스: 등록 후에는 보통 **하루 이내** (하루 1회 이상 방문)

### 6.4 신생 사이트 첫 등록 빠른 인덱스 팁

1. 사이트 등록 → 소유권 확인 (HTML 태그 권장)
2. **글 5-10개 이상** 작성된 상태에서 등록 시도
3. **RSS 제출** (있다면)
4. **사이트맵 제출**
5. 메인 페이지 + 주요 글 5-10개에 대해 **웹페이지 수집** 수동 요청
6. 외부에서 **백링크 1-2개 확보** (블로그·SNS 공유)
7. 2-3주 대기

## 7. 요청 메뉴

### 7.1 웹페이지 수집 (수동 인덱스 요청)

특정 URL을 Yeti가 빠르게 크롤링하도록 요청한다. **하루 할당량 제한**이 있다(보통 도메인별 50건 내외).

권장 사용 시점:
- 신규 글 발행 직후
- 중요한 글 수정 후
- 사이트 이전 직후
- 인덱스되지 않은 URL 발견 시

### 7.2 RSS 제출

**RSS 피드의 역할**:
- 새 글이 발행되면 RSS에 자동 등록
- Yeti가 방문 전에도 새 콘텐츠 정보 푸시
- 뉴스성·일간 갱신 사이트에 강력 권장

**RSS 제출 시 권장 사항**:
- **본문 전체** 포함 (요약만 X)
- **이미지 링크** 포함
- 발행 일자(`pubDate`)와 수정 일자 명확히
- RSS 주소 변경 시 재제출

> **smartdatashop.kr (데이터 저널)**: RSS 필수. Astro의 `@astrojs/rss` 플러그인 사용 가능.

### 7.3 사이트맵 제출

> **네이버는 도메인별 사이트맵 1개만 제출 가능**하다. 모든 콘텐츠를 인덱싱하려면 **사이트맵 인덱스 파일**(여러 사이트맵을 묶은 파일)을 사용한다.

#### 사이트맵 인덱스 구조

```xml
<?xml version="1.0" encoding="UTF-8"?>
<sitemapindex xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <sitemap>
    <loc>https://smartdatashop.kr/sitemap-policy.xml</loc>
    <lastmod>2026-05-05</lastmod>
  </sitemap>
  <sitemap>
    <loc>https://smartdatashop.kr/sitemap-tax-finance.xml</loc>
    <lastmod>2026-05-05</lastmod>
  </sitemap>
  ...
</sitemapindex>
```

이 인덱스 파일 하나(`https://smartdatashop.kr/sitemap-index.xml`)를 네이버에 제출하면 모든 카테고리 사이트맵이 처리된다.

#### 사이트맵 제한
- 단일 사이트맵: **최대 50,000개 URL** 또는 **50MB (압축 전)**
- 인덱스: **최대 50,000개 사이트맵**

### 7.4 웹페이지 검색 제외

특정 URL을 검색 결과에서 제거하고 싶을 때 사용한다.

> ⚠️ **원본 페이지를 삭제하지 않으면 다시 인덱스되어 노출될 수 있다**. 영구 제거하려면 다음을 함께 적용:
>
> 1. 페이지 삭제 (404 또는 410 반환)
> 2. `<meta name="robots" content="noindex">` 추가
> 3. robots.txt에서 차단 (선택)
> 4. 서치어드바이저에서 검색 제외 요청

## 8. 검증 메뉴

### 8.1 URL 검사

특정 URL의 크롤링·인덱스 상태를 확인한다.

표시되는 정보:
- 최근 크롤링 일시
- HTTP 응답 코드
- 인덱스 등록 여부
- 인덱스된 페이지 정보 (제목, 설명, 본문 일부)
- robots.txt 차단 여부
- 메타 태그 (noindex 등)
- 모바일 호환성

인덱스되지 않았다면 **수집 요청** 버튼으로 수동 인덱스 요청 가능.

### 8.2 robots.txt 검증

서치어드바이저에서 `https://smartdatashop.kr/robots.txt`를 가져와 분석한다.

- 파일이 정상적으로 로드되는지
- 문법 오류 여부
- Yeti가 차단되었는지
- 사이트맵이 명시되어 있는지

**수집 요청** 버튼: robots.txt를 수정한 후, 네이버가 빨리 다시 가져가도록 강제 요청.

### 8.3 검색로봇 확인

특정 IP가 진짜 Yeti인지 확인. 위장 봇이 의심될 때 사용.

### 8.4 사이트 간단 체크

URL 하나를 입력하면 다음을 즉시 검증:
- 페이지 타이틀
- 메타 디스크립션
- 메타 키워드
- 오픈그래프 (og:title, og:description, og:image, og:url)
- 캐노니컬 태그
- 모바일 친화성
- HTTPS

> **smartdatashop.kr**: 새 글을 발행할 때마다 이 도구로 즉시 검증하는 것을 권장.

### 8.5 리치 결과 (구조화 데이터)

JSON-LD 구조화 데이터의 유효성을 검증. Google Search Console의 리치 검색결과 테스트와 유사하지만 **네이버가 인식하는 타입은 제한적**이다.

네이버가 부분적으로 인식하는 주요 타입:
- `Organization`
- `WebSite` (사이트링크 검색창 자격)
- `BreadcrumbList`
- `Article` / `NewsArticle`
- `Product` (네이버 쇼핑 연동)
- `Recipe`
- `LocalBusiness`

| **CTR** | 클릭률 (클릭/노출) |

기간별·페이지별·검색어별로 필터링 가능 (단, 검색어 데이터는 **기간 제한**이 있고 Google보다 정보 양이 적다).

### 9.3 콘텐츠 확산 (백링크)

> Google Search Console의 "링크" 보고서와 유사. 우리 사이트로 들어오는 외부 링크 목록.

> **smartdatashop.kr 적용**: 1차 출처(정부 공식 사이트, 통계청 등)를 인용하는 글을 작성하면, 정부 사이트에서 우리를 인용하는 경우는 드물지만 **다른 데이터 분석가들이 인용**할 가능성이 높다.

## 10. HTML 마크업 가이드

### 10.1 네이버가 강조하는 마크업 7가지

1. **페이지 타이틀** (`<title>`) — 페이지마다 고유, 핵심 키워드 포함
2. **메타 디스크립션** (`<meta name="description">`) — 페이지마다 고유, 자연스러운 문장
3. **메타 키워드** (`<meta name="keywords">`) — 5-10개, **네이버는 참고함**
4. **오픈그래프** (`<meta property="og:...">`) — SNS 공유 미리보기
5. **캐노니컬 태그** (`<link rel="canonical">`) — 대표 URL 지정
6. **h1 태그 1개** — 페이지당 단 하나
7. **이미지 alt 텍스트** (`<img alt="...">`) — 이미지 설명

### 10.2 head 영역 권장 템플릿

```html
<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">

  <!-- 페이지 타이틀 (한글 25-32자 권장) -->

  <!-- 메타 디스크립션 (한글 80자 정도 권장) -->
  <meta name="description" content="2026년 5월 종합소득세 신고에 필요한 기한, 공제 항목, 가산세 회피 방법을 1차 출처와 함께 정리했습니다.">

  <!-- 메타 키워드 (5-10개 권장, 네이버는 참고함) -->
  <meta name="keywords" content="종합소득세,2026 종합소득세,종합소득세 신고,1인 사업자 세금,종합소득세 가이드">

  <!-- 작성자 -->
  <meta name="author" content="김준혁">

  <!-- 캐노니컬 -->
  <link rel="canonical" href="https://smartdatashop.kr/policy/comprehensive-income-tax-guide">

  <!-- 오픈그래프 (페이스북·카카오톡·라인 등) -->
  <meta property="og:type" content="article">
  <meta property="og:title" content="2026년 종합소득세 신고 가이드">
  <meta property="og:description" content="2026년 5월 종합소득세 신고에 필요한 기한, 공제 항목, 가산세 회피 방법">

  <meta name="twitter:description" content="2026년 5월 종합소득세 신고에 필요한 기한, 공제 항목, 가산세 회피 방법">

  <meta name="robots" content="index,follow,max-image-preview:large">
```


### 10.4 주요 함정

❌ **하지 말 것**:
- 모든 페이지에 동일한 타이틀·디스크립션 일괄 삽입 (사이트 진단에서 경고)

## 11. 페이지 타이틀·메타 디스크립션·키워드 최적화

### 11.1 페이지 타이틀 (`<title>`)

| 항목 | 권장 |
|---|---|
| **길이** | 한글 **15-32자**, 영문 30-65자 |
| **고유성** | 모든 페이지마다 다른 타이틀 |
| **키워드** | 페이지의 핵심 키워드 1-2개 자연스럽게 포함 |
| **브랜드** | 사이트 이름은 끝에 (구분자: ` - ` 또는 ` | `) |
| **앞쪽** | 가장 중요한 키워드를 앞에 배치 (잘릴 가능성) |
| **자극적** | 과장·낚시 표현 금지 |

### 11.2 메타 디스크립션

| 항목 | 권장 |
|---|---|
| **길이** | **한글 80자**, 영문 150-160자 |
| **고유성** | 모든 페이지마다 다른 설명 |
| **자연스러움** | 1-2개의 자연스러운 문장 |
| **본문 복사 X** | 본문 일부를 그대로 붙여넣지 말 것 |
| **CTA** | "자세히 알아보세요" 등 클릭 유도 표현 가능 |

> 💡 검색 결과의 스니펫은 **메타 디스크립션을 100% 보장하지 않는다**. 검색어의 상황에 따라 본문 일부가 자동 추출될 수 있다. 그러나 좋은 디스크립션은 클릭률을 크게 올린다.

### 11.3 메타 키워드

> ⚠️ **Google은 메타 키워드를 거의 무시**하지만, **네이버는 일정 부분 참고**한다. 그래서 한국어 SEO에서는 여전히 작성하는 것이 안전하다.

| 항목 | 권장 |
|---|---|
| **개수** | 5-10개 |
| **고유성** | 페이지의 실제 내용과 일치 |
| **구분** | 쉼표 `,` 로 구분 |
| **공백** | 쉼표 뒤 공백은 선택 (네이버는 둘 다 인식) |
| **순서** | 중요한 것부터 |

#### 예시

```html
<meta name="keywords" content="종합소득세,2026 종합소득세,종합소득세 신고기한,1인 사업자 세금,종합소득세 공제,종합소득세 가산세">
```

> 💡 **메타 키워드를 도구화하지 말라**. 키워드 스터핑·조작 시그너처로 인식되어 오히려 강등 위험.

### 11.4 키워드 리서치 도구 (한국어)

- **네이버 키워드 도구** (네이버 검색광고 가입 필요): 검색량·경쟁도
- **네이버 데이터랩** (https://datalab.naver.com): 검색어 트렌드, 지역별·성별·연령별
- **블랙키위** (https://blackkiwi.net): 무료 키워드 분석
- **판다랭크** (https://pandarank.net): 블로그·키워드 분석
- **Google Trends** + 한국 필터
- **Google Keyword Planner** + 한국어

## 12. 오픈그래프(og:)·트위터 카드·구조화된 데이터

### 12.1 오픈그래프 (Open Graph)

페이스북이 만든 표준이지만 **카카오톡·라인·페이스북·텔레그램 등 거의 모든 SNS**가 사용한다. 한국에서는 카카오톡 공유가 매우 중요하므로 OG 태그는 필수이다.

#### 핵심 OG 태그

```html
<meta property="og:type" content="article">
<meta property="og:title" content="페이지 제목 (60자 이내)">
<meta property="og:description" content="페이지 설명 (200자 이내)">
<meta property="og:url" content="https://smartdatashop.kr/policy/...">
<meta property="og:image" content="https://smartdatashop.kr/images/...">
<meta property="og:image:width" content="1200">
<meta property="og:image:height" content="630">
<meta property="og:site_name" content="스마트데이터샵">
<meta property="og:locale" content="ko_KR">
```

#### og:type 권장 값

- **`website`** — 일반 페이지, 홈
- **`article`** — 블로그 글, 뉴스 기사
- **`book`**, **`profile`**, **`video.movie`** 등

#### og:image 권장

- **크기**: 1200x630px (페이스북·카카오톡 권장)
- **비율**: 1.91:1
- **포맷**: JPG 또는 PNG
- **용량**: 1MB 이하 권장
- **콘텐츠**: 페이지 핵심을 시각적으로 표현 (텍스트만 큰 이미지보다 시각적 요소 포함이 좋음)

### 12.2 카카오톡 공유 미리보기

카카오톡은 OG 태그를 사용한다. 공유 후 미리보기가 안 보이거나 갱신이 안 될 때:

1. **카카오톡 디버거** (https://developers.kakao.com): 캐시 갱신
2. og:image의 절대 URL 확인 (상대 URL은 인식 안 됨)
3. og:image의 HTTPS 사용
4. 이미지 크기 1200x630 권장

### 12.3 트위터 카드

```html
<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:title" content="페이지 제목">
<meta name="twitter:description" content="페이지 설명">
<meta name="twitter:image" content="https://...">
```

`summary_large_image`는 큰 이미지가 표시되는 형태(권장).

## 13. URL 표준화·캐노니컬·중복 콘텐츠 방지

### 13.1 캐노니컬 태그

```html
<link rel="canonical" href="https://smartdatashop.kr/policy/comprehensive-income-tax-guide">
```

#### 항상 자기 자신을 가리키는 self-canonical 권장

- 파라미터(`?utm_source=...`)에 의한 중복 방지
- `http://` → `https://` 정리
- `www.` → 비-`www.` 정리

### 13.2 도메인 표준화

다음이 모두 다른 URL로 인식될 수 있다:

```
http://smartdatashop.kr/
https://smartdatashop.kr/
http://www.smartdatashop.kr/
https://www.smartdatashop.kr/
```

→ **하나로 통일하고 나머지는 모두 301 리디렉트**.

### 13.3 트래킹 파라미터 처리

UTM 등 트래킹 파라미터가 붙은 URL이 인덱스되지 않도록:

1. **자기 참조 canonical** 사용 (위)
2. URL에 트래킹 파라미터를 넣지 않는 방식 채택 (예: 클라이언트 사이드 처리)
3. 서치어드바이저에서 **검색 제외** 요청

## 14. 모바일 친화성 및 페이지 성능

### 14.1 네이버는 모바일을 어떻게 평가하는가

네이버는 Google처럼 명시적인 "모바일 우선 인덱스(Mobile-First Indexing)"를 공식화하지는 않았지만, **사이트 진단**과 **사이트 간단 체크**에서 모바일 친화성을 평가한다.

평가 항목:
- viewport 메타 태그 (`<meta name="viewport" content="width=device-width, initial-scale=1.0">`)
- 텍스트 크기 (모바일에서 읽기 가능한지)
- 클릭 가능 요소 간격
- 모바일 화면 너비 적합도

### 14.2 페이지 응답 속도

네이버는 **Core Web Vitals** 같은 표준 지표를 직접 채택하지는 않았지만, **응답 속도를 사용자 경험 평가에 반영**한다.

목표:
- **TTFB** (Time to First Byte): 600ms 이내
- **First Contentful Paint**: 1.8초 이내
- **모바일 4G에서 5초 이내 로드**

### 14.3 권장 사항

- HTTPS 필수
- 이미지 최적화 (WebP, 적절한 크기)
- CDN 사용
- 정적 빌드 (Astro·Hugo·Jekyll)
- lazy loading (이미지·iframe)
- 폰트 최적화 (subsetting, font-display: swap)

> Astro + Tailwind 정적 빌드 + Pagefind WASM은 네이버에서도 매우 유리한 출발점이다.

## 15. 유사문서 정책과 어뷰징 방지

### 15.1 유사문서 정책

> 비슷한 콘텐츠는 **원본만 노출**하고 나머지는 검색에서 제외한다.

핵심 원칙:
- 일부만 수정한 복사 = 유사문서로 처리
- 동의어 변환 = 유사문서로 처리
- 자동 paraphrase = 유사문서로 처리
- 유사문서 판단 시스템은 **주기적으로 업데이트**됨

### 15.2 "한 토픽 다중 페르소나 동시 발행" 모델의 위험

> **smartdatashop.kr 메인 + 9개 자매 사이트에 동일 토픽을 동시 발행할 때 가장 큰 위험**이 유사문서 처리이다.

- 동일한 본문을 9개 사이트에 그대로 복사
- 표면적인 동의어 변환만으로 차별화
- AI paraphrase로 양산 확장
- 같은 1차 데이터에서 같은 결론을 같은 방식으로 재생산

#### ✅ 안전한 차별화 방법

## 16. 도메인 정책

### 16.1 Naver 차단 도메인의 특성

> Jun님이 직접 경험한 `smartdata-shop.com`이 Naver 차단된 케이스에서 학습한 것:

#### Naver가 차단할 수 있는 사유 (추정)

1. **이전 소유자의 어뷰징**: Whois에서 과거 어뷰징 도메인이었던 흔적
2. **스팸 DB 등재**: KISA·금감원·국세청 스팸 신고 대상
3. **유사 도메인**: 저명 사이트와 혼동되는 이름
4. **`-` 하이픈 사용**: 일부 패턴에서 의심 시그널 (단정 X, 경향성)
5. **`.com` 외 의심 TLD**: `.xyz`, `.click`, `.top` 등 일부 TLD
6. **만료 도메인 재사용**: 이전 부정적 사용 이력

### 16.2 신규 도메인 사전 검증 6단계 (Jun님 프로세스)

```
1. Naver 등록 가능성 (서치어드바이저에서 "이미 등록됨/스팸 분류됨" 확인)
   → site:domain.kr 으로 네이버 검색
2. Google Safe Browsing 이력 (transparencyreport.google.com)
3. Whois 이전 소유자 (whois.kr, archive.org)
4. 스팸 DB 검색 (KISA 인터넷침해대응센터, 금감원, 국세청)
5. SNS 핸들 가용성 (트위터·인스타그램·유튜브에서 동일 핸들 확인)
6. 한국 상표 검색 (kipris.or.kr)
```

### 16.3 `.kr` vs `.co.kr` vs `.com`

| TLD | 특성 | 추천 용도 |
|---|---|---|
| **`.kr`** | 한국 일반, 짧고 명확 | 메인 도메인 (smartdatashop.kr) |
| **`.co.kr`** | 한국 상업, 전통적 | 대안 (이미 일부 자매 사이트 사용) |
| **`.or.kr`** | 비영리 (제한적) | 공익성 사이트 (awoo.or.kr) |
| **`.com`** | 글로벌, 일반 | 영문/글로벌 타겟 사이트 (iknowhowinfo.com 등) |

> Naver에서는 `.kr` 계열이 유리하다는 가설이 있으나 공식 문서는 없음. 그러나 **사용자 신뢰도 측면**에서 `.kr`이 한국 사용자에게 친숙.

## 17. 네이버 검색 노출이 안 될 때 진단 흐름

### 17.1 1단계 — 등록 상태 확인

```
☐ 서치어드바이저에 도메인이 등록되어 있는가?
☐ 소유 확인이 완료되었는가?
☐ 등록 후 14-16일이 지났는가?
```

### 17.2 2단계 — 인덱스 상태 확인

서치어드바이저 > **사이트 진단**:
- 인덱스 URL 수가 0인가? → 크롤링 자체가 안 됨
- 인덱스 URL은 있지만 노출이 0? → 콘텐츠 품질 또는 어뷰징 의심

서치어드바이저 > **URL 검사**:
- 메인 페이지 URL을 검사 → 응답 코드 + 인덱스 여부 확인

### 17.3 3단계 — 차단 원인 확인

```
☐ robots.txt가 Yeti를 차단하지 않는가?
☐ <meta name="robots" content="noindex"> 가 실수로 들어가 있지 않은가?
☐ 서버가 Yeti의 IP 대역을 방화벽으로 차단하지 않는가?
☐ Cloudflare 등 CDN이 봇 차단 모드인가?
☐ HTTPS 인증서가 유효한가?
☐ DNS가 정상적으로 응답하는가?
```

### 17.4 4단계 — 콘텐츠 품질 진단

**사이트 진단 + 사이트 간단 체크**:
- h1이 2개 이상이면 경고
- 페이지 타이틀 누락 또는 중복
- 메타 디스크립션 누락
- 오픈그래프 누락
- 캐노니컬 누락
- alt 텍스트 누락

**유사문서 의심**:
- 다른 사이트의 콘텐츠를 복사하는가?
- 9개 자매 사이트에 비슷한 글을 동시 발행하는가?
- AI 생성 콘텐츠 비중이 높은가?

### 17.5 5단계 — 수동 인덱스 요청

위 모든 단계가 정상이라면:
1. **요청 > 웹페이지 수집**에서 메인 페이지 + 주요 글 5-10개 수동 요청
2. RSS 재제출
3. 사이트맵 재제출
4. 외부 백링크 1-2개 확보 (블로그 인용, SNS 공유)

### 17.6 6단계 — 그래도 안 된다면

- 네이버 서치어드바이저 **고객센터 문의**
- 일정 기간 (1-2개월) 양질의 콘텐츠를 꾸준히 발행
- C-Rank가 회복되기를 기다림

## 20. 자주 묻는 질문

#### Q. 네이버 검색에 안 떠요. 어떻게 해야 하나요?
A. [§17 진단 흐름](#17-네이버-검색-노출이-안-될-때-진단-흐름) 참고. 등록 후 14-16일 대기, 그 다음에는 안 보이면 robots.txt + 사이트 진단 + 콘텐츠 유사도 점검.

#### Q. RSS와 사이트맵 중 어떤 게 더 중요한가요?
A. **둘 다 제출**하는 것이 가장 안전. RSS는 새 콘텐츠 푸시, 사이트맵은 전체 콘텐츠 지도. 뉴스성 사이트는 RSS가 특히 중요.

#### Q. 메타 키워드를 작성해야 하나요?
A. **네이버는 참고하므로 작성 권장**. Google은 무시하지만, 작성한다고 해해는 없다. 단, **스터핑은 절대 금물**.

#### Q. 9개 자매 사이트에 같은 글을 발행하면 안 되나요?
A. **유사문서 처리되어 모두 노출에서 제외**될 수 있다. 페르소나별 차별화 필수. [§15.2](#152-한-토픽-다중-페르소나-동시-발행-모델의-위험) 참고.

#### Q. 도메인이 네이버에서 차단되었다면?
A. 새 도메인을 사전 검증(6단계)해서 마이그레이션. 차단된 도메인은 외부 노출 유지하고 301 리디렉트만 유지. [§16.1](#161-naver-차단-도메인의-특성), [§19.4](#194-마이그레이션-smartdata-shopcom--smartdatashopkr) 참고.

#### Q. 네이버 블로그·카페에 글을 올리는 게 더 빠른가요?
A. 단기 노출은 빠를 수 있지만, **자체 사이트의 자산**이 되지 않는다. 메인 콘텐츠는 자체 사이트, 일부 발췌·홍보는 네이버 블로그·카페에 분산하는 하이브리드 전략이 권장.
