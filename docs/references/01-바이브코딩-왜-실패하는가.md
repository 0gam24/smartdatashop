# 📘 바이브코딩은 왜 실패하는가 — 참고 요약

> **원전**: https://wikidocs.net/book/19307 (저자: kyung tai park)
> **분류**: 외부 레이어 — 다중 프로젝트 운영 OS
> **smartdatashop.kr 적용 영역**: 자매 사이트 5개 합류 시점, 일일 운영 사이클, 컨텍스트 전환

---

## 1. 핵심 진단

> **"바이브 코딩 실패의 근원은 코드가 아니라 운영 시스템 부재다. 노트와 컨텍스트 관리가 코드 작성보다 먼저 와야 한다."**

- METR 연구: 개발자 체감 생산성 +20% / 측정 생산성 −19%
- 65% 엔터프라이즈 AI 프로젝트 실패 = 하네스(통제 시스템) 결함
- AI 코드 생성 속도가 빨라질수록 사람의 컨텍스트 관리 능력이 병목

---

## 2. 안티패턴 6가지

| 코드 | 패턴 | 위험 |
|---|---|---|
| **A1** | "Accept All" 무한 루프 | AI 결과 검토 없이 모두 승인 → 통제 불능 |
| **A2** | 노트 없이 코드부터 시작 | 의사결정 근거가 휘발 |
| **A3** | progress.md를 사이드 폴더에 방치 | 운영 메타가 코드와 분리 안 됨 |
| **A4** | 컨텍스트 전환 시 처음부터 탐색 | 사이트 갈아탈 때마다 5분+ 헤맴 |
| **A5** | 시간 예산 없이 작업 | "조금만 더..."가 1시간으로 부풀음 |
| **A6** | 도구 결정에 말려 시작 못 함 | 완벽한 도구 선택보다 실행이 우선 |

---

## 3. 적용 가능한 패턴 9가지

### P1. 노트-First 원칙 ★
코드 작성 전 4가지가 노트에 답해져 있어야 함:
1. 무엇을 만드는가? (목적)
2. 왜 만드는가? (이유)
3. 어떻게 만들 것인가? (전략)
4. 어떻게 검증할 것인가? (성공 기준)

### P2. 다중 프로젝트 자기완결 폴더 구조
각 프로젝트(사이트) 폴더가 다음을 모두 포함하면 컨텍스트 전환 5분 이내:
```
[프로젝트]/
├── 개요.md         — 한 문장 요약 + 현재 상태
├── 체크리스트.md   — Phase 진행 현황
├── 설계.md         — architecture / 의사결정 근거
├── 인셉션로그/     — 일자별 작업 기록
├── 에러로그.md     — 디버깅 이력
└── 의사결정.md     — ADR (architecture decision record)
```

### P3. 공통/ 폴더 (횡단 지식)
프로젝트 간 공유되는 지식은 별도 폴더에 발췌:
- SEO 패턴 (모든 사이트 공통)
- 도메인 차단 회피 (smartdata-shop.com 사례)
- 자동화 워크플로우 (n8n)
- AdSense 정책

### P4. 시간 예산 (체크리스트 형태)
| 작업 | 시간 예산 |
|---|---|
| 대시보드 갱신 | 30초 |
| 인셉션로그 작성 | 3분 |
| 컨텍스트 전환 | 5분 |
| 작업 단위 1개 | 30분 (이것보다 길어지면 모듈 분리 잘못된 것) |

### P5. 대시보드 (1 page 메타)
모든 프로젝트 진입의 단일 페이지:
- 모든 프로젝트 현재 상태
- "다음 할 일" 한 줄
- 마지막 작업 일시

### P6. ADR (의사결정 발자국)
큰 변경 시점에 30초만 투자:
```markdown
## YYYY-MM-DD: [결정 제목]
- 컨텍스트: 왜 결정이 필요한가
- 결정: 무엇으로 정했나
- 이유: 왜 이걸 선택했나
- 대안: 무엇을 기각했나, 왜
- 재검토 트리거: 언제 다시 평가할지
```

### P7. 세션 종료 의식 (3분 룰)
세션 끝나기 3분 전에 다시 적기:
- 한 일
- 막힌 곳
- 다음 시작점
- 변경 파일 / 커밋

### P8. 에러로그 자산화
디버깅 시간이 5분 넘으면 에러로그.md에 기록:
- 증상 / 원인 / 해결책
- 같은 에러 두 번째 만나면 1초 안에 해결 가능

### P9. "기록→설계→구현→기록" 사이클
구현 시 노트 작성. 노트가 살아있는 시스템이 되도록.
