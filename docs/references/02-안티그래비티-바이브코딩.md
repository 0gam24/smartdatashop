## 1. 핵심 주장

> **"바이브 코딩의 성공은 코딩 실력이 아니라 논리적 설계 능력에 달려 있다."**
>
> 컨텍스트 엔지니어링 = AI 메모리 공간(RAM)에 무엇을 올릴지 전략적으로 관리하는 시스템 설계.

**Karpathy 비유**: "LLM이 CPU라면, 컨텍스트 윈도우는 RAM이다."

**증명**: OpenAI Codex 100만 줄 / LangChain Terminal Bench 30→5위 / Theo: Cursor +20% / Mitchell Hashimoto: "AI가 실수하면, 구조적으로 그 실수를 반복할 수 없게 만든다"

## 2. 안티패턴 6가지

| 코드 | 패턴 | 위험 |
|---|---|---|
| **A1** | "전체 서비스를 한 번에 만들어줘" | 범위 너무 넓으면 AI도 사람처럼 헤매고 오류 폭증 |
| **A2** | Git 미사용 (비개발자 1순위 실수) | "이전 상태로 되돌려줘" 불가능 |
| **A3** | 코드 먼저, 설계 나중 | DB 구조 나중 변경 = 이미 작성된 코드 절반 재작성 |
| **A4** | 폴더 구조 없이 시작 | AI가 수십 파일을 동시 수정하다 즉시 스파게티 |
| **A5** | 같은 실수 반복 → 룰 파일에 반영 안 함 | 매번 같은 교정 반복 = 시간 낭비 |
| **A6** | 모든 경계 없이 거대한 단일 파일 | 한 파일이 500줄 넘어가면 AI도 사람도 헤맴 |

## 3. 적용 가능한 패턴 9가지

### P1. 성공적인 바이브 코딩 5단계 ★★★
1. **기획 + PRD** — 제품 요구사항 명세서. AI가 참조할 청사진
2. **설계 + 아키텍처** — 데이터 모델, API, 시스템 구성
3. **AI 협업 구현** — 모듈 분할 + SDD(Spec-Driven Development)
4. **지속적 테스트** — 브라우저 에이전트가 사람처럼 QA 수행
5. **배포 + 유지보수** — AI가 모니터링·피드백 분석·개선 제안

> smartdatashop.kr는 **이미 5단계까지 가는 중**. 새 기능 / 새 가이드북 / 새 카테고리 추가할 때 다시 적용.

### P2. 모듈 분리 1줄 기준 ★
> **"이 기능만 따로 테스트할 수 있는가?"**

이 한 줄이 모든 경계의 기준. 독립적으로 동작하고 검증 가능하면 좋은 모듈.

### P3. 구현 순서 (권장)
1. **인증** — 다른 모든 기능 테스트의 전제 조건
2. **핵심 데이터 CRUD** — 서비스 핵심 가치
3. **UI 레이아웃** — 실제 데이터 있어야 UI 검증 가능
4. **부가 기능** — 실시간 동기화, 알림 등

> 정적 사이트(smartdatashop.kr)는 1단계 인증 X. 콘텐츠 → UI → 부가 순서.

### P4. SDD (Spec-Driven Development)
- 명세를 먼저, 그 명세를 기반으로 설계·코드·테스트 작성/검증
- AI 협업 시 혼란/오류를 줄이는 데 가장 효과적
- 큰 기능 추가 전 1~2 페이지 명세가 10시간 절약

### P5. 컨텍스트 엔지니어링 (Karpathy 비유) ★
> **"LLM=CPU, 컨텍스트 윈도우=RAM"**

단순 프롬프트 잘 쓰기를 **넘어서**, AI가 참조할 데이터·문서·코드의 맥락 자체를 설계하는 차원.

**실제 적용**:
- 작업 시작 시 AI에게 어떤 파일을 먼저 읽게 할지 결정
- 큰 작업은 컨텍스트 분할 (탐색 → 구현 분리)
- 불필요한 정보는 컨텍스트에 넣지 않기

### P6. Context Maps — 프로젝트 루트 문서
- `architecture.md` 또는 `file_structure.md`
- 각 폴더의 주요 파일의 역할 명시
- AI가 새 기능 추가 시 기존 구조 파괴하지 않고 적절한 위치에 코드 삽입

### P7. Git 원자 관리
- **원자적 커밋**: 하나의 기능 = 하나의 커밋. 메시지 명확히 ("로그인 기능 추가", "메인 배너 UI 수정")
- **브랜치 전략**: 새 모듈은 별도 브랜치, 성공하면 merge / 실패하면 삭제

### P8. 체크포인트 활용
- 큰 변경 전 항상 git commit으로 안전 포인트
- AI가 로직 꼬이면 복구 가능

### P9. 룰 파일 = 운영자 지침서
- `CLAUDE.md` / `AGENTS.md` / `.antigravityrules` / `.cursorrules`
- "이런 라이브러리는 쓰지 마", "한국어 주석 필수" 등 반영
- **이력 자산화**: 성공한 운영자 프롬프트/구조 지침화 → 다음 작업의 시작 수준 상승

## 4. smartdatashop.kr 즉시 적용 항목

### 이미 적용된 것
- ✅ Astro 컴포넌트 30+개로 모듈 분리 (P2 모듈 분리 기준 충족)
- ✅ `docs/DESIGN`, `docs/PLANNING` 존재 (P6 Context Map 역할)
- ✅ `docs/AGENTS.md` 존재 (P9 룰 파일)
- ✅ GitHub repo + 커밋 이력 (P7)

### 추가로 가치 있는 것
1. **컨텍스트 엔지니어링 의식적 적용** (P5)
   - 큰 작업 의뢰 시 "어떤 파일 먼저 읽어라" 명시
2. **SDD 체크리스트** (P4)
   - 새 가이드북 챕터 추가 시
   - 새 카테고리 페이지 추가 시
   - 30~50편 펄스 일괄 검수 작업 전
3. **모듈 분리 1줄 체크** (P2)
   - 새 컴포넌트 추가할 때마다 자문
