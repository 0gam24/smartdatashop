> **원전**: https://wikidocs.net/book/19414 (저자: 정울피터)
> **분류**: 외부 레이어 보완 — 별도 에이전트 시스템 격리 / 보안
> **smartdatashop.kr 적용 영역**: 6 에이전트 stub 운영 / 멀티 에이전트 분리 결정 / Claude Code vs 자율 에이전트 역할 분담

---

## 1. 핵심 주장

> **"하나의 에이전트로 모든 작업 처리하지 말고, 역할별로 분리하라."**
>
> 멀티 에이전트 격리는 보안·안정성·전문성의 동시 확보. Claude Code는 신규/창의 작업에, Hermes(또는 자율 에이전트)는 반복/지속 운영에 강하다.

**책 19470 Ch.11과의 차이**: 책 19470이 "한 프로젝트 내 멀티 에이전트(Sub-agent)"라면, 이 책은 **"별도 에이전트 시스템 간 격리/통신"**의 관점. 더 큰 단위.

---

## 2. 안티패턴 6가지

| 코드 | 패턴 | 위험 |
|---|---|---|
| **A1** | 한 에이전트에 모든 역할 위임 | 메모리·스킬 충돌, 도메인 최적화 불가 |
| **A2** | Claude Code 없이 처음부터 자율 에이전트(Hermes)에 던지기 | 메모리/스킬 학습 기반 없어 비효율 |
| **A3** | API 키 / 권한 공유 | 한쪽 줄임 = 전체 피해 |
| **A4** | 인터넷 접근 무제한 (default-allow) | prompt injection 취약 |
| **A5** | memory / workspace 백업 안 함 | 에이전트 학습 메모리 = 자산. 손실 시 회복 불가능 |
| **A6** | 격리 수준 일률화 | 모든 에이전트를 같은 격리 수준으로 = 비용 낭비 또는 보안 부족 |

## 3. 적용 가능한 패턴 8가지

### P1. 분리의 3가지 이유 ★
1. **역할 전문화** — 도메인별 메모리/스킬 축적
2. **장애 격리** — 한 에이전트 크래시가 다른 에이전트에 영향 X
3. **보안 경계** — prompt injection 피해 범위 제한

### P2. 4가지 표준 에이전트 구성
| 에이전트 | 역할 | 접근 권한 |
|---|---|---|
| 코딩 에이전트 | 코드 생성/테스트/PR | 로컬 파일, GitHub |
| 운영 에이전트 | 모니터링, 로그 정리 | SSH, Docker, 알림 |
| 리서치 에이전트 | 웹 검색, 문서 요약 | 웹 브라우저, 제한된 파일 |
| 커뮤니케이션 에이전트 | 이메일/캘린더/메시지 | 메시지 API, 캘린더 API |

### P3. 3단계 격리 수준 ★
| 수준 | 방법 | 적합 |
|---|---|---|
| **논리적 분리** | 별도 workspace + config (Hermes profiles 또는 .claude/agents/) | 빠른 테스트, 가벼운 용도 |
| **컨테이너 격리** | Docker 별도, 볼륨/네트워크 분리 | 대부분의 개인 사용자 (권장) |
| **운영 시스템 분리** | 별도 VM/VPS | 보안 중요/장기 운영 |

> **추천 진화 경로**: 논리적 분리 → 권한 커지면 → 컨테이너 격리

### P4. Claude Code vs Hermes Agent 역할 분담 ★★★
| 역할 | Claude Code | Hermes Agent |
|---|---|---|
| 신규 기능 설계 | **강함** | 보통 |
| 프로토타입 | **강함** | 보통 |
| 반복 유지보수 | 세션마다 재시작 | **메모리로 누적** |
| 24/7 자율 실행 | 불가 | **크론, 메시지** |
| 장기 학습 | 세션 간 단절 | **self-improving** |

→ **신규/창의 = Claude Code / 반복/지속 = Hermes (또는 cron + sub-agent)**

### P5. 이전 타이밍 3가지
- MVP/핵심 코드베이스 + 문서(아키텍처, 규칙, 예시) 완성 후
- 반복적/장기 유지보수로 전환 시
- 자율 실행 (크론, 대규모 분석, 배포 자동화) 필요 시

### P6. 에이전트 간 통신 방법
- **파일 기반** (공유 디렉터리) — 가장 단순, 비동기
- **HTTP API** — 동기 호출
- **메시지 플랫폼** (Telegram/Discord/Slack) — 카카오톡 채널도 가능

### P7. 보안 실무 체크리스트 ★
- ✅ API 키를 에이전트 간 **공유하지 않음**
- ✅ 각 에이전트에 **최소 권한만** 부여 (least privilege)
- ✅ 인터넷 접근은 **default-deny + allowlist** 방식
- ✅ 각 에이전트의 memory/workspace **주기적 백업**

## 4. smartdatashop.kr 즉시 적용 항목

### 현재 상태
- **6 에이전트 stub** 보유 (.github/workflows/ + scripts/agents/)
- VSCode + Claude Code in IDE 환경
- Astro+Tailwind 정적 사이트 (1인 운영)
- 1 토픽 → N 페르소나 동시 발행 모델

### 6 에이전트 stub을 P2 4가지 구성으로 매핑 (제안)
| 현재 stub | 매핑 가능 표준 역할 |
|---|---|
| 콘텐츠 작성 에이전트 (writer.mjs) | 리서치 + 코딩 (글쓰기 = 코드의 일종) |
| 콘텐츠 검수 에이전트 (desk-reviewer.mjs) | **별도 분리** (P1 자기 편향 방지 → 책 19470과 일치) |
| SEO 검색 에이전트 (scout.mjs) | 리서치 (웹 분석, 외부 검색) |
| 발행 에이전트 (publisher.mjs) | 운영 (Cloudflare Pages, GitHub) |
| 자매 사이트 동기화 (network-orchestrator.mjs) | 운영 |
| 모니터링 에이전트 (editor.mjs) | 운영 |

### 격리 수준 제안
- **현재 권장: 논리적 분리** (Claude Code의 sub-agent 또는 .claude/agents/ 폴더 활용)
- **컨테이너 격리로 진화 시점**: 자매 사이트 5개 신규 합류 시 (M3~M12)
  - 그때부터 자동 실행 빈도가 늘어나고 외부 API 키도 늘어남

### Claude Code 유지 vs 자율 에이전트 도입 결정

**Claude Code 유지 (현시점 권장)**:
- 신규 기능 설계 (예: "스타트업·창업" 카테고리 신설)
- 새 가이드북 권 시작
- 자매 사이트 신규 합류
- 디자인 시스템 변경

**자율 에이전트 (Hermes 또는 cron + sub-agent) 후보**:
- 일일 펄스 5편 자동 발행 (현재 stub 자동화 대상)
- 자매 사이트 동기화 (정기 cron)
- SEO 모니터링 (정기 감사)

> **단, Hermes 도입은 필수가 아님**. Claude Code의 sub-agent + cron(GitHub Actions)도 비슷한 효과 가능. 사이트는 이미 "7 GitHub Actions 워크플로우 + 6 에이전트 stub" 구성.

### 보안 즉시 적용 가능 (P7)
- ✅ Cloudflare Pages 환경변수의 API 키 — 이미 적용됨
- ⚠️ 각 에이전트 stub에 어떤 권한을 줬는지 명시적 점검 필요
- ⚠️ default-deny + allowlist 정책이 .claude/settings.json에 명시되어 있는지 점검
- ⚠️ memory/workspace 백업 주기 설정 (Git이 일종의 백업이지만 .claude/ 내부 데이터는 별도)

### 하지 말 것 (이미 진행됨)
- 6 에이전트 stub을 한 번에 통합 운영 X (이미 분리됨)
- Claude Code를 통째로 Hermes로 교체 X (창의 작업은 Claude Code가 강함)

## 5. 어떤 작업할 때 이 문서를 참조할지

| 상황 | 참조할 섹션 |
|---|---|
| 6 에이전트 stub을 정리/표준화할 때 | P2 4가지 구성 / 매핑 템플릿 |
| 새 자율 작업 (cron, 24/7 운영) 도입 검토 | P4 Claude Code vs Hermes 분담 표 |
| 보안 점검 시 | P7 보안 4계층 체크리스트 |
| 자매 사이트 5개 신규 합류 시 | P3 격리 수준 진화 / 컨테이너 격리로 전환 |
| 한 에이전트가 여러 일 처리하려고 할 때 | A1 안티패턴 / P1 분리 3가지 이유 |
| 일일 펄스 자동 발행 시스템 만들기 시작 | 책 13-2 "일일 뉴스 브리핑 자동화" 챕터 추가 분석 권장 |
| API 키 관리 방식 검토 | A3 / P7 / 컨테이너 격리 검토 |
| 에이전트가 무한 루프나 폭주할 때 | A4 default-deny / P3 컨테이너 격리 / 명령어 allowlist |
